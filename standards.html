Standard Coding (Backend)

1. การบันทึก Log และระดับของ Log ที่ถูกต้อง

1.1 Info: บันทึกพารามิเตอร์

ควรบันทึกพารามิเตอร์ที่รับเข้ามาในฟังก์ชันก่อนที่จะทำการ _context.QueryAsync หรือ _context.ExecuteAsync

var parameters = new
{
    company = _user.Company,
    listsSelected = request.ListsSelected.ToArray(),
    listsUnselected = request.ListsUnselected.ToArray(),
};

_logger.LogInformation("SQL Parameters: {@parameters}", parameters);

1.2 Info: บันทึกรายละเอียดของกระบวนการ

ใน 1 ฟังก์ชันควรมี Logger อย่างน้อย 3 จุด:

  • Logger Start: อยู่ตอนต้นของ try{...} โดยใช้รูปแบบ: [ชื่อฟังก์ชัน] Start <การทำงาน> <ชื่อ Module>
  • Logger End/Success: อยู่ก่อน return เพื่อยืนยันว่าการทำงานสำเร็จ โดยใช้รูปแบบ: [ชื่อฟังก์ชัน] <End/Success> <การทำงาน> <ชื่อ Module>
  • Logger Error: อยู่ใน catch{...} เพื่อบันทึกข้อผิดพลาด โดยใช้รูปแบบ: [ชื่อฟังก์ชัน] Error <การทำงาน> <ชื่อ Module>
private async Task<IEnumerable<Ddl>> GetEmpType(CancellationToken cancellationToken)
{
    try
    {
        _logger.LogInformation("[GetEmpType] Start Get Employee Type RDMS02");
        // ... code ...
        _logger.LogInformation("[GetEmpType] Successfully retrieved Employee Type RDMS02");
        return result;
    }
    catch (Exception ex)
    {
        _logger.LogError(ex, "[GetEmpType] Error occurred while processing your request Get Employee Type RDMS02");
        throw new Exception(ex.Message);
    }
}

หากเป็นฟังก์ชันที่ handle/handler ให้ใช้ชื่อคลาสแทนชื่อฟังก์ชัน เช่น: [ชื่อคลาส] Start <การทำงาน> <ชื่อ Module>

1.3 Error: บันทึกข้อยกเว้น (Exception)

ใช้ LogError สำหรับบันทึก Exception เท่านั้น เพื่อไม่ให้เกิดความสับสนกับ Log ประเภทอื่น

2. ใช้ AddRangeAsync สำหรับการเพิ่มข้อมูลจำนวนมาก

เมื่อต้องการเพิ่มข้อมูลหลายรายการพร้อมกัน และข้อมูลนั้นตรงกับโครงสร้างของ Entity ให้ใช้ AddRangeAsync เพื่อประสิทธิภาพที่ดีกว่า โดยวนลูปสร้าง List ของ Entity แล้วจึงเรียกใช้ _context.Set<T>().AddRangeAsync(...)

ข้อควรระวัง: ใช้ได้เฉพาะกับการเพิ่มข้อมูล (Insert) เท่านั้น ไม่สามารถใช้กับการอัปเดตข้อมูล (Update) ได้

3. ทุก Method ต้องมี Try-Catch และ Throw new Exception

ทุกๆ Method ควรถูกครอบด้วย try-catch block และใน block catch จะต้องมีการ throw new Exception(ex.Message); ต่อเสมอ เพื่อส่งต่อ Exception ไปยังส่วนกลางสำหรับจัดการ Error

4. การคำนวณทั้งหมดควรทำที่ Backend

เพื่อความปลอดภัยและถูกต้องของข้อมูล ควรหลีกเลี่ยงการคำนวณค่าต่างๆ ที่ฝั่ง Frontend ให้ส่งแค่ข้อมูลดิบหรือ ID ที่จำเป็นมาให้ Backend คำนวณเอง

5. ตรวจสอบฟิลด์ที่จำเป็น (Required Field) ที่ Backend

แม้ว่า Frontend จะมีการตรวจสอบแล้ว แต่ Backend ก็ควรมีการตรวจสอบฟิลด์ที่จำเป็นอีกครั้ง เพื่อป้องกันการยิง Request ตรงๆ ผ่านเครื่องมืออื่น

6. ตรวจสอบความยาวของฟิลด์ให้ตรงกับฐานข้อมูล

สามารถใช้ Data Annotations เช่น [Required] และ [MaxLength(255)] ใน Domain Entities เพื่อกำหนดคุณสมบัติของฟิลด์ให้สอดคล้องกับฐานข้อมูลได้

การใช้ [Required] ไม่สามารถป้องกันค่าที่เป็นช่องว่าง (whitespace) ได้ หากต้องการดักจับช่องว่างต้องเขียนโค้ดตรวจสอบเพิ่มเติม

7. รูปแบบการตั้งชื่อตัวแปร (Naming Conventions)

ประเภทรูปแบบตัวอย่าง
คลาส (Class)PascalCaseLeaveRequest, IUserService
ตัวแปรภายใน MethodcamelCaseleaveType, totalCount
Property / FieldPascalCaseEmployeeId, StartDate
ฟังก์ชัน (Method)PascalCaseSaveData(), ValidateForm()

8. ตรวจสอบชื่อคลาสและ Namespace ให้ถูกต้อง

ชื่อไฟล์ C# ควรตรงกับชื่อคลาสที่อยู่ภายในไฟล์ และชื่อโฟลเดอร์ควรตรงกับชื่อ Namespace

9. ใช้ค่าคงที่ (Constants)

สำหรับค่าที่ไม่เปลี่ยนแปลง (เช่น "Y", "N") ควรประกาศเป็นค่าคงที่ (Constant) ในคลาสที่จัดเก็บค่าคงที่โดยเฉพาะ เพื่อให้โค้ดอ่านเข้าใจง่ายและบำรุงรักษาสะดวก

10. การจัดการไฟล์: ต้องปิด MemoryStream ทุกครั้ง

เมื่อทำงานกับไฟล์ผ่าน MemoryStream, ควรใช้ using statement เพื่อให้แน่ใจว่า Stream จะถูกปิดและคืนหน่วยความจำโดยอัตโนมัติ

XLWorkbook NewWorkbook;
using (MemoryStream ms = new())
{
    request.File.CopyTo(ms);
    ms.Seek(0, SeekOrigin.Begin); // ย้อนกลับไปอ่านตั้งแต่ต้น
    NewWorkbook = new XLWorkbook(ms);
} // ms is automatically disposed here

11. รูปแบบการเขียน SQL Statement

ควรใช้ @"" (Verbatim String Literal) ในการเขียนคำสั่ง SQL ที่มีความยาวหลายบรรทัด เพื่อให้คัดลอกไปทดสอบได้สะดวก

13. การ Log คำสั่ง SQL

ก่อนที่จะรันคำสั่ง SQL ควรมีการ log ตัวคำสั่ง SQL ที่จะถูกรันออกไปก่อนเสมอ เพื่อช่วยให้การดีบักทำได้ง่ายขึ้น

_logger.LogInformation("[List] RDMS02 sql: {sql}", sql.ToString());

14. ตรวจสอบประเภทของไฟล์

ควรมีการตรวจสอบประเภทของไฟล์ที่อัปโหลดเข้ามาว่าเป็นไปตามที่ระบบต้องการหรือไม่ เช่น .pdf, .xlsx, .jpg

15. ตรวจสอบ Array หรือ Collection ก่อนใช้งาน

ก่อนที่จะวนลูป (foreach) หรือเข้าถึงข้อมูลใน Array/Collection, ควรตรวจสอบก่อนว่า object นั้นไม่ใช่ null และมีสมาชิกอยู่ (items.Any()) เพื่อป้องกัน NullReferenceException

16. ตรวจสอบ Null ใน Object ที่มีความเสี่ยง

ก่อนจะเข้าถึง Property ของ Object ที่ได้มาจากการ Query ซึ่งอาจมีค่าเป็น null, ควรมีการตรวจสอบก่อนเสมอ โดยสามารถใช้ Null-conditional operator (?.) ได้

17. Master.cs ไม่ควรมีมากกว่า 1 ไฟล์ต่อ 1 Module

ในแต่ละ Module ควรมีไฟล์ Master.cs เพียงไฟล์เดียวสำหรับจัดการข้อมูล Master ทั้งหมด โดยใช้ switch-case เพื่อแยกการทำงานตามประเภทของข้อมูลที่ร้องขอ




Standard Coding (Frontend)

1. ชื่อ Page และ Component

ตั้งชื่อ Component ให้ถูกต้องและสื่อความหมาย สำหรับหน้าเพิ่มและแก้ไข/รายละเอียด ให้สร้างเป็น Component เดียวกัน โดยใช้ *ngIf='isEdit' เพื่อควบคุมการแสดงผล

2. extends SubscriptionDisposer, implements OnInit และ OnDestroy

ทุกๆ Component ต้อง extends SubscriptionDisposer และเรียก super() ใน constructor หากมีการใช้งาน ngOnInit() ต้อง implements OnInit ด้วย และหาก override ngOnDestroy() จะต้องเรียก super.ngOnDestroy() เป็นบรรทัดสุดท้ายเสมอ

3. ใช้ Constants สำหรับชื่อเฉพาะและ Path

สำหรับค่าคงที่ เช่น URL path ของ API หรือชื่อคอลัมน์ ควรประกาศไว้ใน enum เพื่อให้ง่ายต่อการจัดการและป้องกันการพิมพ์ผิด

4. ใช้ subscribe ใน format (next, error)

เมื่อทำการ subscribe Observable, ให้ใช้ format ที่มี object next และ error เพื่อจัดการกับผลลัพธ์ที่สำเร็จและข้อผิดพลาดอย่างชัดเจน

5. path ใน service ต้องสร้างเป็น constants

เช่นเดียวกับข้อ 3, Path ของ API ที่ใช้ใน Service ควรถูกกำหนดเป็นค่าคงที่ใน enum

6. ไม่ new Date() ด้วยตัวเอง

ให้ยิง API ไปขอเวลาจาก Backend ผ่าน DatetimeService เพื่อให้แน่ใจว่าเวลาที่ใช้เป็นเวลามาตรฐานของ Server ไม่ใช่เวลาจากเครื่อง Client

7. ใช้ DatePipe สำหรับการแปลง Date

สำหรับการจัดรูปแบบวันที่ ให้ใช้ DatePipe ของ Angular โดยต้องเพิ่มเข้าไปใน providers ของ Component และ inject ใน constructor ก่อนใช้งาน

8. ไม่คำนวณที่หน้าบ้าน

หลีกเลี่ยงการคำนวณหรือการตรวจสอบข้อมูลที่ซับซ้อนที่ฝั่ง Frontend ให้ส่งข้อมูลที่จำเป็นไปให้ Backend เป็นผู้ประมวลผลและตรวจสอบ

9. ตรวจสอบขนาดของอาร์เรย์ก่อนประมวลผล

ก่อนเข้าถึงข้อมูลใน Array หรือ Object ควรตรวจสอบก่อนเสมอว่ามีข้อมูลอยู่จริงหรือไม่ เพื่อป้องกันข้อผิดพลาด null/undefined reference แนะนำให้ใช้ฟังก์ชันกลางเช่น isEmpty()

10. ไม่เขียน Style ในไฟล์ HTML

ห้ามเขียน CSS ในไฟล์ HTML โดยตรง (ไม่ว่าจะเป็น <style> หรือ inline style="...") ให้เขียน Style ทั้งหมดในไฟล์ .scss ที่แยกไว้ต่างหาก

11. ไม่ทำ Operation ที่ซับซ้อนใน HTML

หลีกเลี่ยงการเขียนเงื่อนไขหรือ Operation ที่ซับซ้อนใน Template ควรย้าย Logic ไปไว้ในฟังก์ชันของ Component แล้วเรียกใช้จาก Template

ผิด: *ngIf="checked == 'Y' ? true : false"

ถูก: *ngIf="isChecked(checked)"

12. ไม่เขียนโค้ดซ้ำกัน (DRY)

ควรแยกฟังก์ชันที่ทำงานเหมือนกันออกมาเป็นฟังก์ชันกลางที่เรียกใช้ซ้ำได้ และใช้ Component, Service และ Utility ที่มีอยู่แล้วให้เกิดประโยชน์สูงสุด เช่น

  • ใช้ <textbox> แทน <input>
  • ใช้ <areabox> แทน <textarea>
  • ใช้ <number> แทน <input type="number">

13. Check null ทุกครั้งใน Object ที่มีความเสี่ยง

เช่นเดียวกับข้อ 9, ควรตรวจสอบ Object หรือ Array ว่าเป็น null หรือ empty หรือไม่ก่อนใช้งานเสมอ โดยใช้ฟังก์ชันกลางเช่น isEmpty()

14. ครอบ Try-Catch ในฟังก์ชันที่สำคัญ

ฟังก์ชันที่มีการดำเนินการกับข้อมูล เช่น การบันทึก (save), การลบ (delete), การอัปเดต (update) หรือการคำนวณที่สำคัญ ควรถูกครอบด้วย try-catch block เสมอ

15. ใช้การแปลภาษา (i18n)

ห้าม Hardcode ข้อความลงในโค้ดโดยตรง ให้ใช้ระบบการแปลภาษา โดยดึงข้อความจากฐานข้อมูลผ่าน key

ในไฟล์ .html: ใช้ translate pipe เช่น {{ "label.RDMS01.DefineDecoration" | translate }}

ในไฟล์ .ts: ใช้ key กับ Message Service เช่น this.ms.warning("message.PleaseFillSaving")
